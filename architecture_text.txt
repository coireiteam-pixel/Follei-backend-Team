FOLLEI
Complete Scalable Database & Schema Architecture
Domain-Driven Multi-Tenant AI Platform
Version 1.0  •  Architecture Reference
1. Overview & Design Philosophy
Follei follows a domain-driven architecture where every business capability is isolated into its own schema/domain. This prevents giant unreadable tables, impossible joins, scaling bottlenecks, migration nightmares, AI memory chaos, and tenant leakage.
Core Design Goals
Small, focused tables with strong relations
Tenant isolation via tenant_id on every operational table
Event-driven architecture for analytics, replay, and AI learning
Separation of operational vs analytical data
Future analytics warehouse compatibility
Architecture Mental Model
Identity Layer
    ↓
Knowledge Layer
    ↓
RAG Retrieval Layer
    ↓
AI Agent Layer
    ↓
MCP Tool Layer
    ↓
Conversation Layer
    ↓
Analytics & Learning Layer
Multi-Tenancy Principle
Every non-system table carries tenant_id UUID NOT NULL. This enables:
PostgreSQL Row-Level Security (RLS)
Future horizontal sharding
Tenant-level analytics and backups
Enterprise data isolation
Compliance-ready data segregation

7. Critical Architecture Rules
Rule 1: Every Operational Table Has tenant_id
No exceptions. This is the foundation of multi-tenancy, RLS, and future sharding.
tenant_id UUID NOT NULL REFERENCES tenants(id)
Rule 2: Separate Operational from Analytical
conversation_messages must NEVER contain: sentiment, embeddings, metrics, analytics, or feedback. Those belong in dedicated analytical tables.
Rule 3: Events for Everything Important
Every significant action must emit an event to the events table. This enables analytics replay, AI evaluation, debugging, and future ML training.
Rule 4: Embeddings Live in Weaviate
Raw embedding vectors are never stored in PostgreSQL. Only the vector_id reference is stored in chunk_embeddings. This prevents column bloat and enables vector index optimization.
Rule 5: Agent Memory is Tiered
Agent memory is split across three tiers:
short_term: Redis, expires with session, conversation context
mid_term: PostgreSQL agent_memories, persists across sessions, per-customer
long_term: Weaviate memory embeddings, semantic retrieval, global patterns
